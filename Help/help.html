Question: When I run "Rember", I get messages like "FAILURE: 0xfffffdff != 0xffffffff at offset 0x045f3242." reported. What does this mean, and what should I do?

Answer: Rember is an application "wrapper" or GUI for a command line program. This program uses a command-line executable called memtest (this is not developed by Kelley Computing). You can learn more about it by seeing the memtest bundle included with Rember, or by clicking the memtest link above.

When this program reports failures, Rember only displays this in the log window. Future versions will have the abilities to report failures, but as of now (v.0.2.1) the failures will only be logged.

Question (continued): That takes care of how to recognize failures, but how do I take care of the problem?

Answer: If you are not comfortable installing and un-installing hardware in a computer, you should take it to a service technician for diagnosis/repair. If you are comfortable with removing and replacing hardware, the only way (I'm sorry) to locate defective memory chips is by guess and check. This may be updated in the future. You may also attempt to run the Apple Hardware Test CD or DVD that comes with most Macs. This will give more information (if it finds the problem).

Question: Why does Rember eat up so much of the processor's available cycles sometimes, and not at others?

Answer: When the memory tests are running, the Rember program is continuously logging, and (if chosen) filtering "garbage" information (eg. command line progress indicators). This cuts down on processor and memory hogging by Rember. When using this program to test very small amounts of memory (example: 1MB), it will not be efficient. When using the "All" option, this is alleviated. Leaving the "Log" tab selected seems to be the most efficient.
